第 1 页 · 项目总览

Personal Ops Agent

这是一个基于 LangGraph 的混合式个人助理 Agent。我做这个项目的动机有两个： 一是系统性探索 agent、planner、memory 和 tool orchestration 的设计；二是把它做成 一个真实可用的 personal ops assistant，而不是只停留在聊天演示。

  • 任务范围：日程、天气、通勤、Todo、Checklist、日历创建
  • 设计选择：做 bounded assistant，而不是泛化聊天机器人
  • 核心关注：可控性、可观测性、系统级评估

第 2 页 · 系统架构

基于 LangGraph 的混合执行架构

整体系统基于 FastAPI 和 LangGraph。planner node 负责生成结构化执行计划，但系统同时保留 deterministic workflow 作为 baseline 和 fallback，这样既有鲁棒性，也保留了可控性。

执行链路

用户请求
→
/chat API
→
LangGraph
Planner Node
→
Plan Executor
→
Tool Nodes
→
Final Response

为什么这样设计

  • bounded planner：planner 只能从白名单工具中选动作
  • structured state：每一步都有可检查、可测试的状态写回
  • fallback path：planner 不可用时仍然可以走固定 workflow
  • observability：plan、runtime stats、retry、failure 都可见
核心取舍：优先可控、可调试，而不是完全开放式 autonomy。 下一页 →
第 3 页 · 功能范围

面向 personal ops 的聚焦能力

我没有把这个项目做成泛化的对话机器人，而是限定在适合 tool use 和 structured execution 的 personal operations 场景。

日程

读取日历并总结一天或一个时间窗口内的安排。

  • today summary
  • time window parse
  • Google Calendar API

天气

查询当天或周末天气，并返回结构化摘要。

  • today
  • weekend
  • Open-Meteo API

通勤

支持 ETA 查询和出发时间建议。

  • ETA query
  • leave time
  • Google Routes API

任务操作

支持 Todo 创建/读取、出门 checklist、calendar event 创建。

  • todo_create
  • todo_list
  • calendar_create

第 4 页 · 亮点一：Planner

把 planner 作为受约束的 reasoning layer

planner 不是自由生成 agent 行为，而是输出一个 schema 约束下的 `ExecutionPlan`： 包括 intent、ordered actions、reason 和 confidence。这样 LLM 的推理能力被保留， 但执行边界仍然可控。

示例 plan

{
  "goal": "结合下一个日程、天气和通勤信息生成出发建议",
  "intent": "commute_advice",
  "actions": [
    {
      "tool": "schedule_read",
      "args": {
        "date": "today",
        "focus": "next_event"
      }
    },
    {
      "tool": "weather_read",
      "args": {
        "window": "next_event"
      }
    },
    {
      "tool": "commute_plan",
      "args": {
        "destination_source": "next_calendar_event",
        "departure_time": "now",
        "transport_mode": "driving"
      }
    }
  ],
  "reason": "需要读取下一个日程，并结合天气和通勤情况生成出发建议",
  "confidence": 0.9
}

说明：并不是所有 action 都需要复杂参数，但多步 plan 里通常会同时存在两类信息来源： 一类来自显式 args，另一类来自当前 state 或上一步工具输出。

为什么 planner 值得引入

  • 相比纯规则，对边缘自然语言表达更鲁棒
  • 相比 fully open agent，更可控，因为 action space 是有限白名单
  • evaluation 里可以直接比较 planned actions 和 executed actions
  • 支持 hybrid deployment，可以按需开启或关闭 planner
planner 的价值不在于“更自由”，而在于“有边界的语义鲁棒性”。 下一页 →
第 5 页 · 亮点二：上下文设计

上下文设计

这个系统不是简单地把原始文本和 memory 全量塞给模型，而是按阶段拆成不同层次的 context。 planner、execution、response 各自读取不同信息；同时 memory 也被分成图内状态、长期记忆和外部事实来源。

Planner Context

  • user message
  • current time
  • available tools
  • memory snapshot

Execution Context

  • `commute_plan` 读取 location 和 buffer 相关字段
  • `checklist_generate` 读取 weather 和行为偏好
  • node 只消费自己需要的结构化字段

Response Context

  • final output 从 structured state 渲染
  • planner reason 和 runtime stats 可回溯
  • 输出层不再独立“发明”语义
图内工作记忆

Short-Term Memory

  • intent、plan、tool outputs、runtime telemetry
  • 只服务当前请求
  • 支撑 execution 和 final synthesis
持久化用户记忆

Long-Term Memory

  • user profile 和 preferences
  • place aliases
  • behavioral notes，比如开会准备习惯
外部世界

External Memory

  • Google Calendar
  • Todoist
  • weather 和 ETA connectors
上下文和 memory 是分层设计的：不同阶段读取不同信息。 下一页 →
第 6 页 · 亮点三：Evaluation

分层评估 agent pipeline

我专门做了一套 agent system 的 evaluation framework。它不是只看 final answer 对不对， 而是把 routing、planning、execution、response consistency 拆开来评，再结合运行时指标。

评测维度

  • intent routing correctness
  • tool-chain correctness
  • execution fidelity
  • output consistency with structured state
  • required field coverage
  • token、LLM calls、latency、estimated cost

为什么这很重要

  • 支持精确定位错误发生在哪一层
  • 可以做 workflow baseline vs planner 的 A/B
  • 能看清 accuracy 和 cost/latency 的 tradeoff
  • 后续迭代时可复用为 regression suite
evaluation 的价值在于：不仅知道错了，还知道错在哪一层。 下一页 →
第 7 页 · A/B 结果

planner vs baseline

评测结果很清楚地显示了 quality-cost tradeoff：deterministic workflow 更便宜， 而 planner-driven execution 在更复杂的自然语言表达上更稳。

Variant Passed Intent Accuracy Tool Chain Accuracy Avg LLM Calls Avg Tokens
Workflow Baseline 9 / 10 90% 90% 0.4 434
Planner Variant 10 / 10 100% 100% 1.2 1683
  • 最有代表性的收益 case：baseline 把“明天晚上8点安排一个开会”误判成 `schedule_summary`，planner 正确识别成 calendar_create。
这一页是整个项目最核心的证据：planner 不是噱头，而是有量化收益。 下一页 →
第 8 页 · Harness Engineering

Harness Engineering

如果按 agent harness 的核心准则来看，我这个项目已经覆盖了其中大部分，但完成度不完全一样。 这里我更关心的不是“功能做了多少”，而是系统是否具备受约束执行、恢复和观测能力。

Tool

  • planner 只能从固定白名单工具中选 action
  • executor 通过本地 registry 把 tool name 映射到 node
  • 工具调用不是 LLM 直接执行，而是程序按 plan 顺序执行

State Management & Memory

  • LangGraph state 作为 request-scoped working memory，承载 intent、plan 和 tool outputs
  • short-term state、long-term memory 和 external memory 是分层设计的
  • planner、execution、response 读取不同层次的 context

Guardrails & Safety

  • ExecutionPlan、action args 和 memory 都有 schema 约束
  • bounded action space 限制了 agent 的执行范围，不是 fully open agent
  • 当前没有 OS/process-level sandbox，主要依赖 schema、fallback 和 action boundary

Observability & Logging

  • telemetry 记录 token、latency、retry、LLM error 和 trace_id
  • evaluation 可以直接比较 planned actions 和 executed actions，定位 planning / execution 脱节
  • 请求级日志覆盖 router、planner 和 tool execution
这部分更接近 agent harness，而不只是“把功能拼起来”。 下一页 →
第 9 页 · 未来方向

未来方向

下一步我更想做的，不是继续机械地加功能，而是把这个系统从“受约束的单次执行” 推进到“更完整的 agent runtime”。

Runtime 演进

  • 引入更完整的 agent loop，让失败步骤支持 re-plan，而不是只做 fallback
  • 把会话历史加入 context assembly，作为 planner 和 response 的上下文来源之一

Memory 与 Reflection

  • 继续完善 long-term memory，包括更稳的写入策略和更高效的读取方式
  • 引入 reflection：对失败 case 做 self-critique，再把经验沉淀成后续策略
